# Ummat Foundation Donation Manager — Build Specification (Phase 2)

## What this is
A production web app for Ummat Foundation (NGO, India) to manage donations, expenses,
donors, and WhatsApp acknowledgements — plus a public donation page where donors pay
via Razorpay (UPI/cards) and records are created automatically.

A working prototype exists: `ummat-donation-manager.jsx` (in this folder).
**Use it as the reference for features, flows, visual design, and copy.** Reproduce its
look (deep green #0B5C43 / ivory #F6F5EF / gold #C7A028 palette, bottom nav with centre
+ button, receipt sheets, month selector) unless something conflicts with this spec.

## Stack (do not substitute)
- Frontend: React + Vite, mobile-first (max-width 480px layout like the prototype)
- Backend: Supabase (Postgres, Auth, Row Level Security, Edge Functions)
- Payments: Razorpay Checkout (client) + Supabase Edge Function for order creation
  and webhook verification (server)
- Hosting: Vercel
- Excel export: SheetJS (xlsx), same columns as prototype export

## Users & roles
- `admin`: full access — edit/delete records, manage volunteers, settings
- `volunteer`: can create donations/expenses and send WhatsApp thanks; cannot delete
  or edit records older than 24 hours; cannot access settings
- Public donors: NO account. They use the public /donate page only.

Auth: Supabase email/password for admin+volunteers. Roles stored in a `profiles`
table keyed to auth.users, enforced with RLS — never trust a role sent from the client.

## Database schema (Postgres)
```sql
profiles(id uuid pk references auth.users, name text, role text check (role in ('admin','volunteer')), created_at)
donors(id uuid pk, name text not null, phone text not null unique, city text, pan text, lang text default 'en', created_at)
donations(id uuid pk, donor_id uuid references donors, amount numeric not null check (amount > 0),
  date date not null, mode text, purpose text, receipt_no text unique not null,
  source text default 'manual' check (source in ('manual','online')),
  razorpay_payment_id text, notes text, created_by uuid, created_at)
expenses(id uuid pk, amount numeric not null check (amount > 0), date date not null,
  category text, paid_to text, mode text, notes text, created_by uuid, created_at)
settings(key text pk, value jsonb)  -- org name, message templates (en/hi/ur), categories, purposes
receipt_counter: use a Postgres sequence + trigger to generate 'UF-YYYY-NNNN', resets yearly
```
RLS policies:
- donors/donations/expenses: SELECT/INSERT for authenticated admin+volunteer;
  UPDATE/DELETE admin only (volunteer may UPDATE rows they created within 24h).
- Public role has NO direct table access. Online donations are inserted ONLY by the
  Edge Function using the service-role key after webhook verification.

## Features (parity with prototype + additions)
1. Dashboard: all-time balance header, month/year selector + All-time toggle,
   in/out cards, purpose breakdown (flag Zakat separately), activity list.
2. Donations: list, create (search-or-create donor by phone), edit, two-tap delete
   (admin only), receipt sheet with WhatsApp thanks buttons in EN/HI/UR via wa.me links
   (templates from settings; same message text as prototype).
3. Expenses: list, create, edit, two-tap delete.
4. Donors: search, lifetime totals, detail with history, 90-day lapsed list with
   one-tap WhatsApp reminder (EN/HI/UR).
5. Excel export: respects selected period; sheets = Donations, Expenses; same columns
   as prototype (Receipt, Date, Donor, Phone, Amount (INR), Purpose, Mode, PAN, Notes).
6. PDF receipt per donation (org name, receipt no, donor, amount in words, date,
   purpose, "80G details to be added when registration obtained" placeholder line).
7. Settings (admin only): org info, editable purposes/categories, message templates,
   volunteer management (invite by email).
8. Data import: one-time importer that reads the prototype's Excel export
   (Donations + Expenses sheets) and inserts records, creating donors by phone,
   PRESERVING original receipt numbers.

## Public donation page (/donate)
- Fields: name*, mobile (10-digit)*, email optional, amount* (chips 101/501/1100/2100 +
  custom), purpose (same list), PAN optional with note "required above ₹2,000 for
  80G receipt in future".
- Flow (this exact order, no shortcuts):
  1. Client calls Edge Function `create-order` → server creates Razorpay order
     (amount in paise) using key_secret, stores a pending record, returns order_id.
  2. Client opens Razorpay Checkout with the PUBLISHABLE key_id only.
  3. Razorpay webhook `payment.captured` hits Edge Function `razorpay-webhook`;
     verify signature with webhook secret; only then create/update donor + insert
     donation (source='online', receipt generated by DB) via service-role client.
  4. Success screen shows receipt number + a wa.me link so the DONOR can save the
     Foundation's number; admin sees the donation appear in the ledger.
- NEVER insert a donation from the client after checkout callback alone — the
  webhook is the source of truth. Handle duplicate webhooks idempotently
  (unique razorpay_payment_id).

## Security requirements (hard rules)
- key_secret and webhook secret live ONLY in Supabase Edge Function env vars.
- Frontend env contains only Supabase anon key + Razorpay key_id.
- All amounts validated server-side; integer paise; min ₹10, max ₹5,00,000.
- RLS enabled on every table. No table readable by anon role.
- Rate-limit create-order (basic: per-IP counter) to deter abuse.

## Language
- Admin UI: English. WhatsApp templates: EN/HI/UR (copy exact strings from prototype).
- Public /donate page: English + Hindi toggle (two static string sets).
